# Generator Templates

This directory contains Handlebars templates used by plop.js for code generation.

## Template Structure

```
generators/templates/
├── api-action/              # API action handler templates
│   └── action.ts.hbs
├── backend-service/         # Complete backend service templates
│   ├── src/
│   │   ├── index.ts.hbs
│   │   ├── routes.ts.hbs
│   │   └── actions/
│   ├── package.json.hbs
│   ├── tsconfig.json.hbs
│   ├── eslint.config.js.hbs
│   ├── Dockerfile.hbs
│   ├── openapi.yaml.hbs
│   └── .gitignore.hbs
├── documentation/           # Documentation templates
│   └── doc.md.hbs
├── eslint-rule/            # ESLint rule templates
│   ├── rule.js.hbs
│   └── test.js.hbs
└── validation-script/       # Validation script templates
    └── script.sh.hbs
```

## Template Syntax

Templates use Handlebars syntax with custom helpers defined in `plopfile.js`.

### Available Helpers

**String Transformations:**
- `{{camelCase name}}` - Converts to camelCase
- `{{pascalCase name}}` - Converts to PascalCase
- `{{uppercase name}}` - Converts to UPPER_CASE

**Route Helpers:**
- `{{routeToPath route}}` - Converts route to file path
- `{{extractParams route}}` - Extracts parameters from route
- `{{hasParams route}}` - Checks if route has parameters

**Conditionals:**
- `{{#if (eq method 'get')}}...{{/if}}` - Equality check
- `{{#if (contains route ':')}}...{{/if}}` - Substring check

### Example Template

```handlebars
// {{camelCase functionName}}.ts
export function {{camelCase functionName}}() {
  // Generated by plop on {{date}}
  return { serviceName: '{{serviceName}}' }
}
```

## Modifying Templates

1. Edit the `.hbs` file in the appropriate directory
2. Test by running the generator: `bun run plop [generator-name]`
3. Verify the output matches expectations
4. Commit changes with descriptive message

## Adding New Templates

1. Create a new directory in `generators/templates/`
2. Add `.hbs` template files
3. Update `plopfile.js` to add the generator configuration
4. Test the new generator
5. Update documentation

## Best Practices

1. **Keep templates minimal** - Only include essential boilerplate
2. **Use helpers** - Leverage existing helpers for transformations
3. **Add comments** - Include TODO comments where developers need to add code
4. **Follow patterns** - Match existing code style in the repository
5. **Test thoroughly** - Generate code and verify it passes linting
